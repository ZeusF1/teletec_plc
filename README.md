# teletec_plc
OSBB PLC Meter Management — Пряме зчитування архівів
Призначення
Система розроблена для ОСББ (об'єднань співвласників багатоквартирних будинків) що використовують PLC-лічильники MTX (MTX1, MTX3) підключені через маршрутизатор RT4 (STM32 OpenSTLinux).

Проблема: Штатний процес PollReader обходить ~528 лічильників циклічно (~2 за цикл, ~2 хв/цикл), збираючи лише свіжі дані. Глибокий архів (за минулі місяці) отримати неможливо — PollReader скидає дату після обробки. Немає способу оперативно вичитати конкретний лічильник.

Рішення: Модуль прямого зчитування meter_archive.php — надсилає PLC-команди напряму через маршрутизатор, минаючи чергу PollReader. Дозволяє отримати будь-які архівні дані з будь-якого лічильника за вказаний період.

Можливості
⚡ Пряме зчитування архіву (meter_archive.php)
LP (30-хвилинний профіль навантаження) — 48 інтервалів на добу, ~20с/день через PLC
MR (добові показники) — накопичувальні покази по тарифах T1-T4, ~5с/день
Energy (поточні накопичувальні) — загальні покази лічильника на момент запиту
Monthly (місячні показники) — зведення за місяць по тарифах
🔄 Напрямки енергії
A+ споживання — активна енергія з мережі (основний показник)
A- генерація — зворотна активна енергія (сонячні панелі, генерація в мережу)
Q+ реактивна — реактивна імпортована
Q- реактивна — реактивна експортована
💾 Збереження в БД
Зчитані дані зберігаються в штатну БД SQLite (mtxdb.sdb)
LP → таблиця archivo_mhora (з правильним pType: 21/22/23/24)
MR → таблиця indicaciones_mtx1 або indicaciones_mtx3 (pType 1 або 2)
Після збереження дані одразу видимі в графіках UI
🛡️ Захист від перевантаження
Конкурентний лок — одночасно лише 1 пряме зчитування (/tmp/meter_archive.lock)
Авто-чанкування — фронтенд розбиває великі діапазони на частини (3 дні LP / 14 днів MR)
Серверний ліміт — максимум 7 днів LP / 30 днів MR за один HTTP-запит
Не блокує PollReader — використовує окремий лок, PollReader працює паралельно
📊 Інтерфейс (meter_ui.html)
Вибір типу даних, напрямку, діапазону дат
Пресети: 7 / 30 / 60 днів
Живий таймер під час зчитування
Лог прогресу по кожному чанку
Кнопка "Стоп" для зупинки
Масове зчитування кількох лічильників
MR за замовчуванням (найчастіший запит)

Завантаження на RT4
# Завантажити файли
scp www/rt2/meter_archive.php www/rt2/meter_data.php www/rt2/meter_ui.html www/rt2/status_api.php www/rt2/config_api.php root@192.168.100.100:/www/rt2/

# Перезапустити php-fpm (при зависших воркерах)
ssh root@192.168.100.100 "systemctl restart php-fpm"

API приклади
# Добові показники A+ за 2 дні з збереженням
curl "http://RT4:8000/rt2/meter_archive.php?serial=001A790D12263E6E&type=mr&dir=a+&save=1&date_from=2026-03-01&date_to=2026-03-02"

# Генерація A- за тиждень
curl "http://RT4:8000/rt2/meter_archive.php?serial=001A790D12263E6E&type=mr&dir=a-&save=1&date_from=2026-03-09&date_to=2026-03-15"

# 30-хв профіль за 1 день
curl "http://RT4:8000/rt2/meter_archive.php?serial=001A790D12263E6E&type=lp&save=1&date=2026-03-15"

# Поточні накопичувальні покази
curl "http://RT4:8000/rt2/meter_archive.php?serial=001A790D12263E6E&type=energy"

# Переглянути лог
curl "http://RT4:8000/rt2/meter_archive.php?log=1"

Технічні обмеження
Параметр	Значення
Обладнання	RT4 (STM32 OpenSTLinux), 64MB RAM
PHP	8.1.29, php-fpm 3-5 воркерів
БД	SQLite3, WAL mode
PLC час відповіді	LP ~20-120с/день, MR ~5-90с/день
Глибина архіву	LP ~60 днів, MR ~12 місяців (в пам'яті лічильника)
Одночасних зчитувань	1 (конкурентний лок)
Max за запит	LP 7 днів, MR 30 днів
